这对当今的环境意味着什么？摆脱平均表现从未如此重要。虽然脱离群体存在风险，但留在群体中的风险（至少在表现方面）要大得多。最近几年的平均回报率简直糟透了（看Cambridge Associates的数据，DPI、RVPI、TVPI都在全面下滑）。有人可能会说基金仍处于 J 曲线阶段，但我认为这并不能解释我们看到的优秀表现者和卓越表现者之间的差异。

“公有制还是私有制都不重要，核心问题是垄断”​

真正的重点不是国有化和私有化，而是究竟有没有市场化。

国内对于国有化和私有化这个问题的争论是长期存在严重偏差的，容易把重点放在所有制层面，但实际上所有制是一个次要的因素，真正的决定性因素是究竟有没有实现市场化。

市场化的对立面不是公有制，而是垄断，罪恶来自垄断而非来自所有制。不论处于什么所有制条件下，只要一家独大，排斥竞争，旱涝保收，统购统销，整个生态系统都会迅速烂的干干净净。

当供应商只有一个的时候，这个供应商究竟是公有的还是私有的就一点都不重要了，继而，产品质量和服务水平也不重要了。

公有制下垄断，会变成“禁止打骂顾客”的供销社，私有制下垄断，会变成寡头经济，而市场化是目前已知的避免垄断的唯一有效途径。

在所有制上撕扯没有任何意义，官僚主义，僵化保守，低效落后，人浮于事，唯利是图，大到不能倒，这些情况既可以以大公司病和寡头经济的形式出现在私有制条件下，也同样可以以票证经济，供销社和统购统销的形式出现在公有制条件下。

所以针对所有制展开争论纯属找错了目标，真正的问题应该是如何消灭垄断，保持竞争，坚持市场化，寡头对于任何一个社会来说都是一场彻头彻尾的灾难，不论是私有寡头还是公家寡头，是寡头就得死，天底下就不该有大到不能倒，资不抵债了就该倒，天王老子也得倒。

公有制之所以在历史上屡屡败在私有制手下，是因为公有制能够依靠公权力以极低的成本和极高的效率迅速完成垄断，并因为垄断而迅速摧毁整个生态系统。

公权力既是公有制的受益者，又是规则的制定者，因此天然倾向于在制定规则的过程中排斥竞争，消灭市场，通过垄断来让自身收益最大化。

而私有制要想形成垄断，要么通过超强的产品力形成人无我有，只此一家式的商业壁垒（如英伟达），要么拥有压倒性的体量优势，依靠迅速调集资源来在局部竞争中压倒竞争对手（如企鹅），要么把持主要行业渠道（如芝加哥粮食期货交易所），要么官商勾结借助公权力来形成垄断（如俄罗斯的能源寡头），形成垄断的成本非常之高，难度非常之大，很多垄断本身就是激烈竞争后的最终结果，没有公有制那种一纸行政命令直接结束战斗的便利性。

也正是这种形成垄断时在成本和效率上的差距，导致公有制更容易形成垄断，并因为垄断而迅速腐朽，从而在与私有化的竞争中败下阵来。

左人要想给公有制摇旗呐喊，与其把注意力放在攻击私有制上，不如想想办法，如何在公有制条件下保持市场竞争，防止垄断，允许破产。最起码，也应该尝试提高公有制形成垄断的成本，防止垄断以及紧随垄断之后必然产生的腐朽。

skiplist 与平衡树、哈希表的比较

skiplist 和各种平衡树（如 AVL、红黑树等）的元素是有序排列的，而哈希表不是有序的。因此，在哈希表上只能做单个 key 的查找，不适宜做范围查找。所谓范围查找，指的是查找那些大小在指定的两个值之间的所有节点。

在做范围查找的时候，平衡树比 skiplist 操作要复杂。在平衡树上，我们找到指定范围的小值之后，还需要以中序遍历的顺序继续寻找其它不超过大值的节点。如果不对平衡树进行一定的改造，这里的中序遍历并不容易实现。而在 skiplist 上进行范围查找就非常简单，只需要在找到小值之后，对第 1 层链表进行若干步的遍历就可以实现。

平衡树的插入和删除操作可能引发子树的调整，逻辑复杂，而 skiplist 的插入和删除只需要修改相邻节点的指针，操作简单又快速。

从内存占用上来说，skiplist 比平衡树更灵活一些。一般来说，平衡树每个节点包含 2 个指针（分别指向左右子树），而 skiplist 每个节点包含的指针数目平均为 1/(1-p)，具体取决于参数 p 的大小。如果像 Redis 里的实现一样，取 p=1/4，那么平均每个节点包含 1.33 个指针，比平衡树更有优势。

查找单个 key，skiplist 和平衡树的时间复杂度都为 O(log n)，大体相当；而哈希表在保持较低的哈希值冲突概率的前提下，查找时间复杂度接近 O(1)，性能更高一些。所以我们平常使用的各种 Map 或 dictionary 结构，大都是基于哈希表实现的。

从算法实现难度上来比较，skiplist 比平衡树要简单得多。

如果单纯比较性能，跳跃表和红黑树可以说相差不大，但是加上并发的环境就不一样了，如果要更新数据，跳跃表需要更新的部分就比较少，锁的东西也就比较少，所以不同线程争锁的代价就相对少了，而红黑树有个平衡的过程，牵涉到大量的节点，争锁的代价也就相对较高了。

- trie 树空间消耗大，但是不会冲突，哈希查找占用空间小，但是因为哈希冲突，会有冲突的问题
- trie 树和哈希查找的效率。trie 树的插入和查询都是 O(m) 的 (m 是待插入/查询的字符串的长度)，复杂度是稳定的，哈希查找理论复杂度是 O(1) 的，但是哈希查找实际受限于哈希函数的性能，复杂度不确定。我们通常使用复杂度稳定的算法。
- 按照我的理解，搜索树（或者说平衡树）的使用场景基本上已经被哈希表和 skiplist 覆盖了，哈希表虽然不支持范围查找，但是在不需要这些场景的情况下，哈希表的时间复杂度 O(1) 就已经完胜了。而 skiplist 相比于搜索树，内存占用更高，时间复杂度相当，还支持更方便的范围查找。是吗？如果不是，请举出反例
- skiplist 本身也可以处理有序数据啊，并且绝大部分情况下 skiplist 的内存占用都更低，只有子节点内的查找，搜索树的内存占用更低吧 # 是的，skiplist 本身也可以处理有序数据。相比于搜索树，skiplist 在绝大部分情况下具有更低的内存占用。在 skiplist 中，每个节点都包含一个指向它们在不同层级的下一个节点的指针。这种结构使得在查找某个节点时，可以通过跳过一些节点来加快搜索速度。而在搜索树中，每个节点都包含左子树和右子树的指针，因此需要更多的内存空间来存储这些指针。所以，总体来说，skiplist 的内存占用要比搜索树更低。
好的，让我们从时间复杂度、空间复杂度、是否有序排列、范围查找、内存占用和算法实现难度这几个方面来比较Skip List、Red-Black Tree、Hash Table和Search Tree（B树和B+树）。

1. **时间复杂度**：
   - **Skip List**：平均O(log n)，最坏O(n)。
   - **Red-Black Tree**：O(log n)。
   - **Hash Table**：平均O(1)，最坏O(n)。
   - **Search Tree**：O(log n)。

2. **空间复杂度**：
   - **Skip List**：O(n)，因为需要额外的空间来维护多级索引。
   - **Red-Black Tree**：O(n)，每个节点需要存储额外的平衡信息（如颜色标记）。
   - **Hash Table**：O(n)，需要预分配空间以减少冲突，但实际空间使用取决于哈希函数和负载因子。
   - **Search Tree**：O(n)，每个节点可能存储多个键和子树指针。

3. **是否有序排列**：
   - **Skip List**：是，元素按照顺序排列。
   - **Red-Black Tree**：是，元素按照顺序排列。
   - **Hash Table**：否，元素无序。
   - **Search Tree**：是，元素按照顺序排列。

4. **范围查找**：
   - **Skip List**：支持，可以快速进行范围查找。
   - **Red-Black Tree**：支持，可以进行中序遍历来进行范围查找。
   - **Hash Table**：不支持，因为元素无序。
   - **Search Tree**：支持，可以进行范围查找。

5. **内存占用**：
   - **Skip List**：相对较高，因为需要额外的索引层。
   - **Red-Black Tree**：相对较低，但比哈希表高，因为需要存储平衡信息。
   - **Hash Table**：相对较低，但取决于负载因子和哈希冲突。
   - **Search Tree**：相对较高，因为每个节点可能存储多个键。

6. **算法实现难度**：
   - **Skip List**：中等，需要正确处理层级跳跃。
   - **Red-Black Tree**：较高，需要处理复杂的旋转和重新着色操作。
   - **Hash Table**：较低，但需要设计一个好的哈希函数来减少冲突。
   - **Search Tree**：中等，需要正确处理节点分裂和合并。

总结：
- **Skip List** 和 **Red-Black Tree** 都支持有序排列和范围查找，但Skip List在最坏情况下的性能可能退化。
- **Hash Table** 提供了最快的平均查找时间，但不支持有序排列和范围查找，且在最坏情况下性能可能退化。
- **Search Tree**（特别是B+树）在数据库和文件系统索引中非常有效，支持有序排列和范围查找，但内存占用相对较高。
- 在算法实现难度上，**Hash Table** 最简单，而 **Red-Black Tree** 最复杂。

Youtube’s Anti-adblock and uBlock Origin - And a Dinosaur


我之前也好奇像 adblock 这种浏览器插件是如何屏蔽掉 Youtube 这类网站嵌入在网页或者视频播放器里的广告。

起初我以为广告插件是简单粗暴地屏蔽掉了某些 DOM 元素，但事实比我想象的要复杂得多。

在上面这篇文章里，作者讨论了视频网站嵌入广告的技术路线，以及反广告插件是如何通过技术手段使其失效的。

其基本原理如下：

1、视频网站会在前端请求一个 API，获取面向当前所播放视频相关的广告信息，这些信息是通过 JSON 格式进行传递的，广告信息就在其中某个数组里。

2、广告屏蔽插件会重写覆盖 JS 引擎的 JSON.parse 函数，使得视频网站前端代码在执行解析之后，获得一个空的广告信息数组。（想出这个点子的人真是个天才）。

不过就像军备竞赛一样，双方都在非常认真地升级，近期 Youtube 实在忍无可忍，通过“先发送虚假请求并判断信息是否被篡改”的方式，主动识别出安装了这类插件的浏览器，并弹出窗口严词要求用户关掉该插件。

跨平台方案​

跨平台开发该不该用Flutter？2023年版Flutter全面解析_哔哩哔哩_bilibili 视频不错，很喜欢这种娓娓道来的感觉

看这个视频时，我想到了两个问题：

  • 这几种移动端跨平台方案，从现在的视角来看是一地鸡毛。大家还是转回移动端原生语言的方案。但是那个时候我们是真心认为“js统一大前端”的。
  • dart和ts都是对js类型支持的解决方案，那么为啥dart在和ts的竞争中失败了？

这个视频也一下子把我拉回到6、7年前我刚刚入行的时候，那个时候移动端余温尚存，各种移动端跨平台方案层出不穷，大家都在讨论各种“三端一致”的方案。实话说如果不是这个视频，这部分记忆都要淡忘了。几种主流方案：

  • 移动端打包浏览器跨平台（比如Hbuilder, AppCan, Cordova）
  • react-native, uniapp(相当于vue的react-native)
  • flutter

这三种方案各有优劣，第一种方案是最早的跨端方案，主打快糙猛，很快就被淘汰了。后两种方案，react-native以及uniapp这种就是用js来调用各种原生移动端语言的API的，可想而知，一定会存在适配问题，并且会很慢（因为走了两层，并且无论是js还是swift, OC, java, kotlin性能都不算很好）。flutter则使用dart实现，dart可以理解为某种js方言，flutter则是吸取了以上教训，直接自己（通过chrome的render engine, skia）实现各种移动端的UI，那么这种方案的问题也同样可想而知，生态问题！如果使用flutter就不能使用。

所以通过这两种解决方案，可以看到是左右为难，用粘合层来实现就会有性能问题，用类似flutter自己实现的方案则无法使用移动端现有生态。并且二者都会存在很多适配问题。

至于为啥到最后一地鸡毛，大概还是因为跨平台本身可能还是个伪需求。某种开发者认为很重要，技术管理者认为一般重要，但是老板认为不重要的东西。并且确实以上三种方案从开发角度，也都离原生应用差的有点远，可能会有各种延迟、不适配之类的bug。总结来说就是老板认为不重要，开发者认为重要但是实现不了。最后一地鸡毛也就不奇怪了。

比较golang map和redis map​

数据结构

  • 相同：内部两个哈希表，用于扩容，但Go中叫做buckets和oldbuckets，Redis中是一个数组，大小为2
  • 不同：层次不同。 参见上面的图，Redis第二层存储了子表的信息，第三层作为子表，存储的是实际数据的地址；Go实际只有三层。这导致两种实现后续功能的差异，如：是否支持缩容。
  • 不同：size、used存储方式。Go在顶层结构中存储了B字段，表示有2^B个bucket，并存储了count字段表示已有数据个数；Redis在第二层(每个字表信息)中
  • 不同：k-v排列方式。Go：8key+8val作为一个bucket，后面可以链式挂接更多overflow的bucket，Redis：key+val作为一个dictEntity，后面可以链式挂接更多dictEntity

哈希方式

  • 相同：根据不同类型，调用不同的hash方法后，求余得到索引
  • 不同：Redis：得到bucket索引后即得到bucket数据，二Go还需要再根据tophash->key的查找

冲突解决方式

  • 相同：拉链法
  • 不同：Redis的链表直接存在每个数据(dictEntity)后，Go由8个k-v组成一个bucket，然后再挂接overflow bucket

rehash

  • 相同：装载因子的概念
  • 相同：渐进式扩容 下文详细描述
  • 不同：触发扩容的时机 Go在插入新key时检测装载因子和拉链长度，Redis在增删查改时都会检查是否需要rehash
  • 不同：触发扩容的条件 Go：bucket内(可以存储8个k-v)的平均个数超过6.5或单个bucket的overflow超过bucket数会进行扩容；Redis：每个key平均存储了一个数据，则进行扩容；每个key平均存储了不到0.1个数据，则进行缩容
思考：

假如Go的数据主要集中在一个bucket里，其overflow很长，其他bucket的数据很少，这时应该触发rehash吗？
这要分两种情况：

如果该bucket包括overflow中的数据量比较满，那么map整体的数据量也接近6.6/bucket了，rehash扩容后，hash后的key被重新打散，数据会被重新分配到其他bucket中；

如果该overflow中的位置比较空，rehash后bucket数量不变，该bucket中的数据被重新依次填入到新的bucket中，空位消失，overflow数量也就减少了

渐进式扩容

  • 相同：第一次先分配空间，后面再渐进搬迁
  • 不同：Go只在增删操作时搬迁，Redis在增删查改操作时都会进行渐进搬迁操作

缩容

  • 不同：Go只保存了当前的bucket size，新bucket一定是旧bucket大小的两倍，不支持缩容。Redis的缩容：两个子表记录了自己的大小，缩容即扩容的逆过程

代码组织方式

  • 不同：Go编译器会做很多trick的处理，比如给bmap填充真实的字段和数据，而Redis使用C编写，代码一目了然
